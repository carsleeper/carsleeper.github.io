문제풀이 개요

문제에서 제공해주는 서버에 접속하면 다음과 같은 화면이 나온다.

또한 서버구조를 보면 frontend-side를 static과 templates폴더가, backend-side를 app.py가 담당하고 있음을 알 수 있다.

서버의 하위 url에 대한 내용들을 하나씩 분석하면 문제의 단서를 얻을 수 있다.

def read_url(url, cookie={"name": "name", "value": "value"}):    
    cookie.update({"domain": "127.0.0.1"})
    try:
        service = Service(executable_path="/chromedriver")
        options = webdriver.ChromeOptions()
        for _ in [                                                               
            "headless",
            "window-size=1920x1080",
            "disable-gpu",
            "no-sandbox",
            "disable-dev-shm-usage",
        ]:
            options.add_argument(_)
        driver = webdriver.Chrome(service=service, options=options)               
        driver.implicitly_wait(3)
        driver.set_page_load_timeout(3)
        driver.get("http://127.0.0.1:8000/")
        driver.add_cookie(cookie)      
        driver.get(url)                    
    except Exception as e:
        driver.quit()
        print(str(e))
        # return str(e)
        return False                  
    driver.quit()
    return True

함수의 내용이 좀 긴데 실행과정은 다음과 같다.

  1. 14번 이전 라인들에서 Selenium Webdriver를 이용하기 위한 기본 설정들을 한다.
  2. 14번라인에서 Web driver를 실행한다.
  3. 18~19번 라인에서 인자로 받은 쿠키를 추가한 후, 인자로 받은 url에 get요청을 보낸다.
  4. 만약 get 요청에 성공하면 True를, 예외가 발생하면 False를 리턴한다.

이 함수의 내용을 요약하면, 인자로 받은 url에 인자로 받은 cookie를 server-side에서 get 요청한다.

check_csrf 함수

def check_csrf(param, cookie={"name": "name", "value": "value"}):
    url = f"http://127.0.0.1:8000/vuln?param={urllib.parse.quote(param)}"
    return read_url(url, cookie)

역시 특정 url에 라우팅되지 않은 일반 함수이다. 로컬호스트에서 /vuln 페이지에 param 인자를 전달하는 url을 read_url함수에 넘긴다. 리턴 값이 read_url함수 이므로, 최종적으로 url에 접속하면 True, 예외가 발생하면 False를 리턴한다.

@app.route("/vuln")
def vuln():
    param = request.args.get("param", "").lower()
    xss_filter = ["frame", "script", "on"]
    for _ in xss_filter:
        param = param.replace(_, "*")
    return param

param 인자의 value 값을 받아와서 그 값이 frame, script, on이 아니면, 문자열을 클라이언트로 리턴해준다.
여기서 중요한 점은 클라이언트에서 받는 response packet에는 Content-type이 html로 명시되어 있기 때문에, <img src=”Attack URL”> 와 같은 태그를 리턴받는다면 element node로 해석되어 Client 사이드에서 “Attack URL”에 get 요청을 보낼 수 있다는 점이다.

@app.route("/flag", methods=["GET", "POST"]) 
def flag():
    if request.method == "GET":
        return render_template("flag.html")
    elif request.method == "POST":
        param = request.form.get("param", "")
        if not check_csrf(param):
            return '<script>alert("wrong??");history.go(-1);</script>'
        return '<script>alert("good");history.go(-1);</script>'

위와 같이 flag.html에서는 파라메터를 POST 방식으로 입력받고 있다. 이를 참고 해서 backend 코드를 보자.
get 요청일 경우 클라이언트에게 웹 페이지를 제공해주고, post 요청일 경우 인자를 받아서 check_csrf에 넘겨주고 있다. 만약 read_url 함수에서 http://127.0.0.1:8000/vuln?param=userinput 에 접속한다면 True를 리턴해서 7번라인의 if문을 거치지 않고 good을 alert할 것이고, 접속 과정중 예외가 발생한다면 wrong??을 alert할 것이다.

flag -> check_csrf -> read_url 의 과정을 정리하면 다음과 같다.
사용자가 입력한 인자 value를 localhost/vuln에 server-side에서 넘겨주게 될 것이다.

@app.route("/admin/notice_flag")
def admin_notice_flag():
    global memo_text
    if request.remote_addr != "127.0.0.1":
        return "Access Denied"
    if request.args.get("userid", "") != "admin":
        return "Access Denied 2"
    memo_text += f"[Notice] flag is {FLAG}\n"
    return "Ok"

중요한 함수이다. 이전까지 봐왔던 함수들은 모두 admin_notice_flag함수에서 flag값을 뽑아내기 위한 조건들을 충족시킬 밑바탕이었다.
이 함수에서는

  1. localhost에서 접속하고
  2. userid 인자의 value가 admin이어야

memo_text에 flag값을 추가해준다. 1번 조건은 flag -> check_csrf -> read_url 의 과정에서, 2번조건은 flag함수의 param 입력값에서 만족시킬 수 있다.

Solution

flag의 입력란에 넣을 텍스트는 최종적으로 vuln에 들어갈 인자값이므로, <img src=”/admin/notice_flag?userid=admin”> 을 입력하면 read_url 함수가

  1. localhost에서
  2. userid=admin 인자값으로

vuln페이지에 get요청을 보내게 되고 vuln페이지는 localhost로 <img src=”/admin/notice_flag?userid=admin”> 를 리턴하게 된다.
그럼 localhost에서 <img src=”/admin/notice_flag?userid=admin”>를 받고 element node로 해석하여 해당 링크에 get 요청을 하게 된다.
admin_notice_flag함수는 두가지 조건을 충족하였으므로 memo_text에 flag를 추가하게 되고, 최종적으로 memo 페이지에서 flag를 확인할 수 있다.

느낀점

  1. 중요한 부분에만 집중해야 한다.
    예를 들어서, flag함수가 check_csrf함수를 호출할 때, cookie 인자 값은 넘겨주지 않는데, check_csrf함수의 cookie 인자에 대해 너무 깊게 고민하지는 말자는 의미이다. 어떤게 중요하고, 어떤게 중요하지 않은 지는 결과론적으로 알 수 있는 내용이지만, 최대한 빨리 분간해낼 수 있는 능력을 기르자.

  2. 행위가 어느 side에서 이뤄지는가를 정확하게 파악해야 한다.
    이 문제를 처음 풀었을 때, 단순하게 내 클라이언트에서 /vuln?parm=<img src=”/admin/notice_flag?userid=admin”> 로 접속하면 server-side에서 src링크에 get요청을 보내서 그 내용을 내 클라이언트에 전해주는 줄 알았다. 그래서 내 브라우져에서 <img> 인자를 포함한 vuln url 에 접속하면 flag가 뜰 줄 알았다.
    그런데 사실은, server-side에서는 단순히 “<img src=”/admin/notice_flag?userid=admin”>“라는 텍스트만 내 클라이언트에 리턴해주고, src url에 get 요청을 보내는 건 내 클라이언트였다는 것이다. 따라서 localhost에서 접속해야 한다는 조건을 충족하지 못하게 된다.
    어떻게 보면 정말 기본적인 부분이지만, 정말 세세하게 엄밀히 따지고 들어가야 위와 같은 실수를 하지 않을 것 같다.
